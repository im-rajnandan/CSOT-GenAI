# Week 2 Project Submission

For my Week 2 project, I made **ResearchBot**. It can search the web, open and read web pages, find academic papers, and then use all that information to give an answer. I used Serper for web search, `requests` for fetching pages, and the AlphaXiv MCP server for finding and reading papers. I decided to use the openrouter free model `owl-alpha` for LLM.

At first I kept a simple CLI version because it helped me debug the agent loop properly. Later I made the main interface using Textual, so now it works like a full-screen terminal app. The Textual UI can stream the answer as it is generated, show when tools are being used, keep the chat history, and also has shortcuts for clearing the screen, resetting the conversation, clearing the input box, and quitting the app.

The main logic is inside the `ask_agent` function. When the user asks something, I add the question to the message history and send the full history along with the available tools to the model. If the model wants to call a tool, I first save that tool-call message, then run the function it asked for, and then add the result back as a tool message. After that, I call the model again with the new information. This keeps repeating until the model finally gives a normal answer instead of asking for more tools. I also added `MAX_AGENT_STEPS` so that if the model gets confused, it does not keep searching or fetching forever.

For reading web pages, I first check the website’s `/llms.txt` file if it exists. Then I fetch the page and try to clean it using `trafilatura`. If that does not work well, I fall back to `markdownify`. Trafilatura is useful because it removes a lot of useless page stuff like menus and ads, while markdownify is useful when the structure of the page matters, like headings, links, and tables. I also truncate the page text before sending it back to the model, because one big webpage can take up too much context. I made the tools return small success or error dictionaries, so if something fails, the agent can still continue instead of the whole program crashing.

Some of the annoying bugs appeared in the TUI. I had added `Ctrl+K` to clear the history, but it did not work when the input box was focused because the Input widget caught the key first. I fixed that by making the shortcuts priority bindings. The screen also became messy for a moment whenever a paper tool ran. I found that `mcp-remote` was printing connection and shutdown logs directly to the terminal while Textual was using the same screen. I fixed it by hiding those process logs while still returning real tool errors normally. Streaming was another challenge because tool-call names and JSON arguments sometimes arrived in separate chunks, so I had to join the pieces before running the tool and avoid showing duplicate final messages.
